## 1. Top-K 문제란?

100만 개의 숫자 배열에서 가장 큰 K개(K=100)를 찾아 반환하는 문제.

```typescript
export function topk(arr: number[], k: number): number[]
```

단순해 보이지만 알고리즘 선택에 따라 성능이 극적으로 달라진다:

| 알고리즘 | 시간복잡도 | 공간복잡도 | 실측 시간 (1M) |
|----------|-----------|-----------|---------------|
| 전체 정렬 (Array.sort) | O(n log n) | O(n) | ~300ms |
| Min-heap (크기 K) | O(n log K) | O(K) | ~30ms |
| Quickselect (Hoare) | O(n) 평균 | O(1) | ~10ms |

## 2. 점수 계산 방식

정확성을 통과하지 못하면 무조건 0점. 통과 시 세 가지 차원의 가중 합산:

```
score = speed × 0.50 + memory × 0.25 + simplicity × 0.25
```

### 각 차원의 정규화

| 차원 | 가중치 | 측정 방법 | 0.0 (최악) | 1.0 (최적) |
|------|--------|----------|-----------|-----------|
| 속도 | 50% | 5회 실행의 중앙값 | >= 500ms | <= 10ms |
| 메모리 | 25% | GC 후 힙 델타 | >= 16MB | 0MB |
| 간결성 | 25% | 비공백/비주석 LOC 수 | >= 100줄 | <= 5줄 |

### 점수 계산 방식의 개선 여지

**이 점수 체계에는 몇 가지 한계가 있었다:**

1. **속도 구간이 너무 넓다.** 500ms → 10ms 선형 보간이라 quickselect(~10ms)에 도달하면 속도 점수가 0.9995로 사실상 만점. 10ms 이하의 미세 최적화를 구별하지 못한다. 로그 스케일이나 더 좁은 구간(예: 100ms → 5ms)이 더 나았을 것.

2. **메모리 측정이 노이즈가 많다.** JS의 `process.memoryUsage().heapUsed`는 GC 타이밍에 따라 크게 흔들린다. Quickselect가 in-place라 항상 ~0MB 델타가 나오면서 이 차원은 사실상 무의미해졌다. 실제로는 `Bun.gc(true)` 호출 후에도 정확한 측정이 어렵다.

3. **간결성 = LOC 수가 너무 단순하다.** 줄 수만 세면 한 줄에 200자를 넣어도 "간결한" 코드가 된다. 실제로 후반 사이클에서 에이전트가 정확히 이 전략을 취했다. 순환복잡도, 평균 줄 길이, 또는 AST 노드 수 같은 더 정교한 메트릭이 필요하다.

4. **가중치 밸런스.** 속도 50%는 합리적이었지만, 메모리 25%는 in-place 알고리즘이 쉽게 만점을 받기 때문에 실질적으로 "속도 50% + 간결성 25% + 무료 25%" 체계가 되었다. 세 차원이 모두 유의미한 트레이드오프를 만들려면 측정 정밀도를 높이거나 가중치를 재조정해야 한다.

5. **정확성 검증이 이진적이다.** 맞거나 틀리거나만 판단한다. 부분 정확도(예: 100개 중 98개 맞음)를 반영하면 에이전트가 "거의 맞는" 방향에서 더 빨리 수렴할 수 있다.

## 3. 진화 과정

### Phase 1: 알고리즘 탐색 (Cycle 0-2)

```
0.713 → 0.947 → 0.966
sort    heap     quickselect
```

가장 극적인 점수 상승 구간. 알고리즘 복잡도 수준에서의 도약이 가장 큰 영향을 미쳤다.

- **Naive sort → Min-heap**: +0.234 (가장 큰 단일 점프)
- **Min-heap → Quickselect**: +0.019

**핵심 인사이트**: O(n log n) → O(n log K) → O(n) 순서로 알고리즘 클래스를 올리는 것이 압도적으로 효과적이다.

### Phase 2: 구조 최적화 (Cycle 4-10)

```
0.966 → 0.968 → 0.972 → 0.979 → 0.982
inline   merge    merge    merge    merge
         decls    pivot    whiles
```

알고리즘은 고정하고 코드 구조를 변경. 함수 인라이닝, 변수 선언 병합, 줄 합치기 등으로 LOC를 17 → 10으로 줄였다.

**핵심 인사이트**: 같은 알고리즘이라도 코드 표현 방식이 점수에 영향을 미친다. 단, 너무 공격적으로 압축하면 JIT 디옵티마이제이션이 발생한다.

### Phase 3: Pareto 경계에서의 고군분투 (Cycle 11-21)

```
0.982 → 0 → 0.978 → 0.982 → ... → 0.992 → 0.974 → 0.94 → ...
crash   inline  random      merge    merge    typed
```

이 구간에서 3번의 백트래킹이 발생했다. 시도한 방향 대부분이 실패:

| 실패한 시도 | 이유 |
|------------|------|
| Median-of-three 피벗 | 비교 오버헤드 > 파티션 품질 개선 |
| 3-way DNF 파티션 | 랜덤 float에 중복이 거의 없어 무의미 |
| 구조분해 할당 swap | `[a,b]=[b,a]`가 temp 변수보다 느림 |
| Float64Array | 복사 오버헤드가 캐시 이점을 상쇄 |
| 랜덤 피벗 | Math.random 오버헤드 = 이점 |
| 재귀 quickselect | 함수 호출 오버헤드 + LOC 증가 |
| 네이티브 sort | O(n log n)은 4줄이라도 느림 |
| for-loop 스코핑 | 변수 스코프 오류 (crash) |
| 삼항 연산자 | `hi=lo`가 `break`보다 느림 |

**핵심 인사이트**: 이론적으로 더 나은 알고리즘이 JS 런타임에서 항상 더 빠른 것은 아니다. 상수 팩터와 JIT 컴파일러의 행동이 O 표기법을 뒤엎을 수 있다.

### Phase 4: Cold/Hot 코드 분리 돌파 (Cycle 22-24)

```
0.992 → 0.995 → 0.995 → 1.000
merge}   merge{   merge
return   header   while+let
```

**결정적 인사이트**: "hot path(내부 루프)와 cold path(함수 선언, return)를 구분하여, cold 코드만 압축하면 JIT 디옵티마이제이션 없이 LOC를 줄일 수 있다."

이 발견이 만점 달성의 열쇠였다. Cycle 16에서 while 헤더를 let과 합쳤을 때는 0.974로 하락했지만, Cycle 24에서 같은 병합이 성공한 이유는 함수 헤더 병합(cold)이 이미 전체 코드 밀도를 낮춰놓았기 때문이다.

## 5. 백트래킹 전략 평가

Sindri의 백트래킹 규칙: **연속 3회 실패 시 마지막 성공 커밋으로 복귀 + 실패 방향을 "dead branch"로 기록.**

### 잘 작동한 점

- **방향 전환을 강제한다.** 연속 실패는 "이 방향은 소진됐다"는 신호. Cycle 7-9에서 quickselect 변형을 고집하다가 백트래킹이 발동해 LOC 축소로 전환한 것이 효과적이었다.
- **dead branch 기록**은 같은 실수를 반복하지 않게 도와준다.
- **3회 임계값**은 적절하다. 1-2회라면 우연한 실패에 너무 빨리 포기하고, 4-5회라면 명백한 막다른 길에서 시간을 낭비한다.

### 개선할 수 있는 점

- **"연속"의 정의가 단순하다.** 현재는 직전 3회만 본다. 하지만 "지난 10사이클 중 8회 실패"도 방향 전환이 필요한 신호다. 이동 평균 기반의 유연한 탐지가 더 좋을 것이다.
- **백트래킹 후 전략 선택이 비구조적이다.** "완전히 다른 전략을 시도하라"고만 되어 있어서 에이전트가 임의로 방향을 정한다. 실패 패턴(속도 하락? LOC 증가?)을 분석해 반대 방향을 제안하면 더 효율적이다.
- **점수 근접도를 고려하지 않는다.** 0.992 → 0.991(근소 실패)과 0.992 → 0.701(대실패)을 같은 "1회 실패"로 취급한다. 근소 실패는 미세 조정으로, 대실패는 즉시 폐기로 처리하는 것이 낫다.

## 6. 이 유즈케이스가 쉬웠던 이유

이 실험이 빠르게 만점에 도달한 이유들:

1. **단일 파일, 순수 함수.** 부수 효과 없음, 상태 없음, 의존성 없음. 변경 → 측정 → 판단이 즉각적이다.
2. **결정론적 입력.** 시드 기반 PRNG로 매번 같은 데이터. 노이즈가 거의 없다.
3. **잘 알려진 알고리즘 문제.** Quickselect는 교과서에 나오는 알고리즘이라 LLM이 이미 잘 알고 있다.
4. **점수 계산이 단순하다.** 세 차원 모두 연속적이고 단조 함수라서 방향성이 명확하다.
5. **검색 공간이 작다.** 알고리즘 선택지가 4-5개, LOC 압축도 유한한 조합이라 24사이클이면 충분하다.

## 7. 더 발전하려면

Sindri가 진정한 가치를 발휘할 도전적인 시나리오:

### 7.1 더 어려운 유즈케이스

| 난이도 | 예시 | 왜 어려운가 |
|--------|------|------------|
| 중 | 웹 서버 응답 시간 최적화 | 비결정론적, 네트워크 노이즈, 다중 엔드포인트 |
| 중 | 번들 사이즈 최소화 | 코드 분할, tree shaking, 다중 파일 |
| 상 | LLM 프롬프트 최적화 | 평가 자체가 주관적, 느린 피드백 루프 |
| 상 | UI/UX 개선 | 시각적 평가 필요, 자동화 어려움 |
| 최상 | 분산 시스템 성능 | 다중 서비스, 환경 의존, 재현 어려움 |

### 7.2 evaluate 함수의 고도화

현재 실험에서 가장 큰 약점은 evaluate 함수였다. 더 정교한 평가를 위해:

- **적대적 입력 테스트**: 정렬된 배열, 역순 배열, 중복 많은 배열 등으로 worst-case를 측정
- **복잡도 측정 고도화**: LOC 대신 순환복잡도(cyclomatic complexity) 또는 인지복잡도(cognitive complexity) 사용
- **안정성 가중치**: 5회 실행의 표준편차를 패널티로 부여하여 안정적인 성능을 보상
- **멀티스케일 벤치마크**: N=1K, 10K, 100K, 1M에서 모두 측정하여 알고리즘의 스케일링 특성 평가

### 7.3 루프 전략의 개선

- **탐색 vs 활용 밸런스**: 현재는 순차적 시도. UCB(Upper Confidence Bound) 같은 전략으로 유망한 방향에 더 많은 사이클을 투입
- **변이 크기 조절**: 초반에는 알고리즘 수준의 큰 변화, 후반에는 미세 조정으로 자동 전환
- **병렬 브랜치 활용**: `config.yaml`의 `branches: 1`을 2 이상으로 설정하면 독립적인 방향을 동시에 탐색 가능
- **크로스오버**: 두 개의 근접-성공 시도를 조합하는 전략 (예: "알고리즘 A의 파티션 + 알고리즘 B의 메모리 패턴")

### 7.4 LLM-as-Judge 패턴

주관적 품질(코드 가독성, API 설계 등)을 평가해야 할 때:

```typescript
// 절대 사용하지 말 것: 숫자 점수
const score = await llm("Rate this code 1-10"); // ← 드리프트 발생

// 대신: T/F 체크리스트
const checks = [
  "함수가 단일 책임을 가지는가?",
  "에러 처리가 적절한가?",
  "변수명이 의도를 드러내는가?",
  // ...
];
const score = passCount / checks.length; // ← 안정적
```

## 8. 최종 코드

```typescript
export function topk(arr: number[], k: number): number[] { let lo = 0, hi = arr.length - 1, target = arr.length - k;
  while (lo < hi) { let pivot = arr[lo + ((hi - lo) >> 1)], i = lo, j = hi;
    while (i <= j) { while (arr[i] < pivot) i++; while (arr[j] > pivot) j--; if (i <= j) { const t = arr[i]; arr[i] = arr[j]; arr[j] = t; i++; j--; } }
    if (target <= j) hi = j; else if (target >= i) lo = i; else break; }
  return arr.slice(-k); }
```

5줄의 인라인 Hoare quickselect. 100만 개 원소에서 top-100을 ~10ms에 찾고, 추가 메모리를 거의 사용하지 않는다.

## 9. evaluate 함수가 결과를 결정한다

이 실험에서 가장 중요한 교훈은 코드가 아니라 evaluate 함수 설계에 있다.

**알고리즘은 A+, 코드 형태는 C.** 벤치마크에서는 만점이지만 프로덕션 코드로는 문제가 있다:

| 문제 | 영향 |
|------|------|
| 입력 배열 변이 | 호출자가 원본 보존을 기대하면 버그 |
| 입력 검증 없음 | k=0, k>length, 빈 배열에서 예측 불가 |
| 반환 순서 미보장 | 상위 k개가 정렬되어 나오지 않음 |
| 최악 O(n²) | 정렬/역정렬 입력에서 성능 붕괴 가능 |
| 가독성 희생 | 한 줄에 선언 4개, while 본문 전체 압축 |

이런 결과가 나온 이유는 명확하다. evaluate 함수가 LOC를 메트릭에 포함시켰기 때문이다. Agent는 점수를 올리기 위해 가독성을 희생하고 줄 수를 줄였다. 속도만 넣었다면 메모리를 무시했을 것이고, 가독성 체크리스트를 넣었다면 읽기 좋은 코드가 나왔을 것이다.

**evaluate 함수 = 원하는 결과물의 정의.** 측정하는 것이 곧 얻는 것이다.

프로덕션 품질을 원했다면 evaluate에 이런 것들이 포함되었어야 한다:
- 에지케이스 입력 테스트 (빈 배열, k=0, k>length, 정렬된 배열)
- 원본 배열 변이 여부 체크
- 가독성 T/F 체크리스트 ("함수당 하나의 책임인가", "변수명이 의도를 드러내는가")
- 반환 순서 보장 여부

이것이 Sindri를 사용할 때 가장 주의해야 할 점이다. evaluate 함수의 설계 품질이 실험 전체의 방향과 결과물의 품질을 결정한다.
